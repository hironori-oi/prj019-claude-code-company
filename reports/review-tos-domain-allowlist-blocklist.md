# PRJ-019 OpenAI ToS 対象分野 ホワイトリスト / ブラックリスト 原案

- 案件: PRJ-019「Clawbridge（仮）」 — Open Claw を自律オーナーとする AI 組織ハーネス基盤
- 部署: レビュー部門（品質管理）
- 作成日: 2026-05-03
- 作成者: Review Agent (claude-code-company)
- 版: v1
- 関連決裁: DEC-019-009〜013（CEO 決裁、2026-05-02）

## 0. 文書の位置づけ

### 0.1 目的
- Clawbridge が「ニーズ判定 → 完全自動公開」する DoD パスにおいて、**OpenAI ToS（および Anthropic AUP / 一般法令）に抵触する分野を自動 reject**、**安全に完全自動公開できる分野を allow**、**HITL ゲート要 のグレーをケースバイケース判断** にするための分類基準を確定する。
- 必須コントロール v2 の G-11「公開可能アプリ allowlist」を**実装可能な分類辞書**まで具体化する。

### 0.2 対象とする ToS 規定
オーナー貼付の OpenAI 使用ポリシー全文より、**「機密性の高い領域での重大な意思決定の、人間の確認のない自動化」を禁止**する 13 分野が起点:

> 重要インフラ / 教育 / 住居 / 雇用 / 金融活動・与信 / 保険 / 法律 / 医療 / 重要な行政サービス / 製品安全コンポーネント / 国家安全保障 / 移住 / 法執行

加えて使用ポリシーは「**人を操る／欺く目的、人権の行使を妨げる目的、人の弱みを悪用する目的、人の教育を受ける能力や重要サービスを利用する能力を妨げる目的**」を全面禁止しているため、これらを横断条項として加味する。

### 0.3 設計方針
- **3 レイヤ分類**: ブラックリスト（即棄却）/ ホワイトリスト（完全自動公開許容）/ グレー（HITL 必須）。
- **「アプリ機能 × 影響度」で判定**: 同じ分野でも「単なる情報提示」と「自動意思決定」では判定が分かれる。
- **保守的判定（in dubio pro reo の逆）**: 迷ったらブラックリスト寄せ、CEO 個別許可で例外昇格。

## 1. ブラックリスト（完全自動公開禁止、人間確認必須）— 13 分野

### B-01. 重要インフラ
| カテゴリ | 具体例（NG） | グレーゾーン境界 | 例外条件 |
|---|---|---|---|
| 制御系 | 電力管理、水道、ガス、エネルギーグリッド、原子力、ダム、信号機・交通管制、通信網制御、データセンター冷却 | 「停電情報の単純可視化」「料金シミュレータ」は OK | 例外昇格は不可（一律 NG） |
| 監視系 | SCADA 系統、産業用 IoT 制御、ライフライン異常検知の自動アクション | 「異常検知アラートの人間通知のみ」は HITL ゲートでグレー | 人間最終承認 + 出力が**通知のみ**で制御アクション無し |
| 物流・交通 | 自動運転制御、列車運行、航空管制、海運運行 | 「乗換案内」「混雑可視化」は OK | 例外昇格不可 |

**理由**: 1 件の誤判定で人命・国民生活に直接波及するため、Clawbridge ニーズ判定 → 自動公開の DoD では絶対に通さない。

## 2. ホワイトリスト（完全自動公開許容）— 6 分野

**前提**: ホワイトリスト分野でも「ホワイトリスト内でも避けるべきこと」を満たす場合は**グレー降格**。

## 3. グレーゾーン（HITL ゲート必須、ケースバイケース判断）

### G-Top-1. マッチング系（出会い、求人、不動産紹介）
**境界条件**: 出会い系は出会い系規制法 (異性紹介事業届出義務) 直撃、求人マッチングは B-04 雇用境界、不動産紹介は B-03 住居境界。
**HITL ゲート仕様**:
- ニーズ判定スコアリングで「マッチング」「出会い」「紹介」「求人」「不動産紹介」キーワード検出 → **CEO 個別承認**を要求
- アプリ実装段階で「双方向のやり取り or 自動マッチング判定」を含むなら**ブラックリスト降格**
- 「単純な検索ディレクトリ」（一方向の情報提示のみ）なら HITL 通過後ホワイトリスト扱い
**判断基準**: ① 個人 vs 個人の取引仲介を含むか ② 法的責任の主体は誰か ③ 規制業に該当するか の 3 点で CEO が判定。

### G-Top-2. 個人情報を扱うもの（顧客管理、CRM、メール配信）
**境界条件**: 個人情報保護法・GDPR・特定電子メール法 (オプトイン規制) 直撃。
**HITL ゲート仕様**:
- 「顧客」「メール配信」「CRM」「会員管理」「ユーザー登録」キーワード検出 → **CEO 承認 + プライバシーポリシー策定必須**
- メール配信機能ありなら**事前同意取得フローの実装証跡**を Review 検収
- EU ユーザーアクセス想定なら**GDPR 対応コスト**を CEO 判定 (DPO 設置・SCC 等)
**判断基準**: ① 個人データ取得の同意フロー ② 削除・開示請求対応 (個情法 33 条) ③ 漏洩時の通知体制 (個情法 26 条) ④ 越境移転 (GDPR 45-49 条) の 4 点。

### G-Top-3. 決済を含むもの（小規模 EC、サブスク、寄付）
**境界条件**: PCI-DSS (カード情報非保持なら緩和)、特商法 (通信販売表示義務)、資金決済法 (前払式・送金)、景表法 (二重価格・優良誤認)。
**HITL ゲート仕様**:
- 「決済」「課金」「サブスク」「EC」「寄付」「投げ銭」キーワード検出 → **CEO 承認 + 特商法表示策定必須**
- カード情報を自前で持つ実装は**自動 reject**（Stripe / Square 等の PSP 経由必須）
- 寄付 / 投げ銭は**特定商取引法・古物営業法・資金決済法**該当判定を Review が個別実施
**判断基準**: ① 決済手段 (PSP 利用か) ② 特商法表示要件充足 ③ 暗号資産関連は資金決済法該当性 ④ 寄付は税務 (寄付金控除) の取扱い。

### G-Top-4. ユーザー間取引（マーケットプレイス、フリマ、シェアリング）
**境界条件**: 古物営業法 (中古売買)、宅建業法 (不動産)、薬機法 (医薬品)、特商法、資金決済法 (エスクロー)、プロバイダ責任制限法。
**HITL ゲート仕様**:
- 「マーケットプレイス」「フリマ」「中古」「個人間売買」「シェア」キーワード検出 → **CEO 承認 + 仲介リスク評価**
- 古物・酒類・医薬品・不動産のいずれかを含むなら**業法直撃**で**ブラックリスト降格**を原則
- 「情報マッチングのみ・取引はオフライン」なら HITL 通過後ホワイトリスト扱い
**判断基準**: ① 取引対象が業法対象か ② 仲介者として責任を負うか (取消権・補償) ③ 通報・モデレーション体制 ④ 利用規約の責任分界。

## 4. ニーズ判定スコアリング関数の修正仕様

### 4.1 既存スコアリングへの追加レイヤ
PM v2 の「HN / Product Hunt / GitHub Trending スコアリング」関数に**ジャンル分類器レイヤ**を前置する。

```
入力: ニーズ候補テキスト (タイトル + 本文 + タグ + 上位コメント要約)
処理:
  1. ジャンル分類器 (LLM, claude -p with --json-schema)
     → 出力: {category: "B-01"〜"B-13" | "W-01"〜"W-06" | "G-Top-1"〜"G-Top-4" | "UNKNOWN", confidence: 0..1, reasons: [...]}
  2. 分類結果に基づくスコア乗数:
     - "B-XX" (ブラックリスト) → score = 0 (即棄却), 監査ログに reason 全文記録
     - "G-Top-X" (グレー) → score = min(score, 0.3), HITL ゲート (Slack 承認) 発動
     - "W-XX" (ホワイトリスト) → score 通常計算継続
     - "UNKNOWN" or confidence < 0.7 → グレー扱い (G-Top-* と同等 HITL)
出力: {final_score, hitl_required, blacklist_reason}
```

### 4.2 分類器のプロンプト要件
- **System prompt 必須記載項目**: 13 ブラックリスト分野の定義 (本書 §1) + 6 ホワイトリスト分野 (本書 §2) + 4 グレー (本書 §3) + 横断禁止条項 (操作・欺瞞・人権侵害・政治運動・学問不正)
- **`--json-schema`**: `{category: enum, confidence: number, reasons: string[], gray_subcategory: string?, blacklist_subcategory: string?}` でハードチェック
- **逐語証拠提示要求**: `reasons[]` には「ニーズ候補テキスト中の該当箇所引用 + ToS 該当条文」を最低 1 件含めること

### 4.3 精度評価方法（False Positive / False Negative）
| 指標 | 定義 | 目標値 | 計測方法 |
|---|---|---|---|
| **FN-Black**（最重要） | ブラックリスト該当ケースを許容してしまう率 | **< 1%** | 月次、Review が手動サンプル 100 件を独立分類して classifier 判定と照合 |
| **FP-Black** | ホワイトリスト該当ケースをブラックリストに誤分類 | < 10%（緩め） | 同上 |
| **FN-Gray** | グレーをホワイトリストに誤分類 | < 5% | 同上 |
| **Confidence Calibration** | confidence < 0.7 のケースの実際の誤分類率 | 30% 以上で UNKNOWN として HITL に回せれば OK | 月次 |
| **総合判定** | FN-Black が 1 件でも検出されたら**即停止 + 分類器再学習 / プロンプト改修** | — | Review が判定 |

### 4.4 分類器の運用ルール
- **更新権限**: CEO 決裁 (本書 §1〜§3 改訂と連動)
- **更新トリガー**: ① ToS 改定、② 新分野出現 (例: 生成 AI 規制法成立)、③ 誤判定 (FN-Black ≥ 1) 発生、④ 月次レビューでの新パターン発見
- **更新頻度**: 最低四半期 1 回 (G-V2-10 ToS 半年再評価サイクルと同期)
- **A/B テスト禁止**: 自動公開判定の精度劣化リスクが致命的なため、新版分類器は**並行 shadow run** で 2 週間検証後に切替

## 5. ホワイトリスト / ブラックリストの運用ルール

### 5.1 更新ガバナンス
| 項目 | 詳細 |
|---|---|
| **更新権限** | CEO 決裁 (本書改訂は DEC-019-XXX で記録、Review が起案、Research が ToS 整合確認) |
| **更新トリガー** | ① OpenAI 使用ポリシー改定、② Anthropic AUP / Consumer Terms 改定、③ 国内法令改正 (個情法・特商法・薬機法等)、④ 誤判定発生、⑤ 新規アプリカテゴリの台頭 |
| **更新頻度** | 最低四半期 1 回。半年再評価サイクル (G-V2-10) と統合 |
| **緊急更新** | 重大事故 (誤公開・行政指導等) 発生時は 24 時間以内に分類器停止 + CEO 緊急決裁 |

### 5.2 監査ログ要件
- ニーズ判定スコアリングの**全件**について `category / confidence / reasons / hitl_required / blacklist_reason / final_score` を append-only 監査ログに記録
- ブラックリスト棄却ケースは 5 年間保持 (証跡)、ホワイトリスト通過ケースは 90 日保持
- HITL ゲート発動ケースは**承認者 ID + 承認理由 + 承認時刻**を必須記録

### 5.3 例外昇格プロセス（グレー → ホワイト）
- グレーゾーン G-Top-1〜4 の特定案件を「HITL 通過後ホワイトリスト扱い」とする例外昇格は**CEO 個別決裁** (DEC-019-XXX で記録)
- 例外昇格条件: ① 業法該当性が Review/法務観点で問題なし、② プライバシー・決済・仲介の各リスクが個別審査済、③ 1 年後の継続審査義務あり

### 5.4 例外降格プロセス（ホワイト → グレー / ブラック）
- ホワイトリスト分野でも「ホワイトリスト内でも避けるべきこと」(本書 §2) に該当する実装が判明した場合、**自動 reject**
- 降格判定は**コードレビュー段階の Review** が起票、CEO 承認で確定
- 既に公開済みの場合は**即時公開停止 + インシデント手順** (G-12 副作用ゼロ証明と同等)
