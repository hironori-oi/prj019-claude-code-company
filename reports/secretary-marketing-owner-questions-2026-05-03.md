### Q-Mkt-01. PATTERN-006 / PATTERN-007 番号衝突回避

### Q-Mkt-03. 表現比重（harness 40% / org 25% / cost 20% / ToS 15%）の Owner 承認

- **背景**: `marketing-portfolio-reflection-design.md` §2.1 訴求軸 4 軸の重み付け表で提示。AI 自律を主役に据えない理由（CLAUDE.md 事業方針「AI 感を出さないクリーンなデザイン」+ Open Claw 上流 personal AI assistant 再ポジション）と整合。
- **CEO 推奨案**: 表記比率のまま採用。CEO コメント: 「中小企業向け受託訴求に最適、harness 工学を最前面が他社差別化」。
- **Owner 判断項目**: A) 提示比重で承認／ B) 比重微調整（指定）／ C) 4 軸再構成。
- **判断材料**:
  - A 案 pros: Marketing / CEO 整合済、即着手可。cons: 比重根拠は内部判断のみ。
  - B 案 pros: Owner の販売戦略を直接反映。cons: 比重変動で Hero / S2〜S7 全コピー再起草、5〜7 営業日遅延。
  - C 案 pros: 訴求軸の根本見直し。cons: §3 LP 7 セクション設計が白紙化。
- **回答が遅れた場合の影響**: §3 LP コピー案（§3.2 S1〜S7）の凍結が遅れ、Web 運営の Next.js 実装着手が後ろ倒し。

### Q-Mkt-04. Heading 案 A vs B vs C の Owner 最終確定

- **背景**: `marketing-portfolio-reflection-design.md` §2.4 で 3 案提示。A「自社プロダクトを、4 週間で安全に検証する。」が Marketing 推奨。同 §7-1 で CEO 判断項目として明示。
- **CEO 推奨案**: A 採用。B / C は技術ブログ本文中の小見出しに転用。CEO コメント: 「A は中小企業発注検討者（45% 想定読者）に過度な技術臭を抑え刺さる」。
- **Owner 判断項目**: A / B / C のいずれか（自由記述による D 案も可）。
- **判断材料**:
  - A: 落ち着き重視、発注検討者向け、技術臭最小。
  - B「自律運用の前に、運用境界を引く。」: やや技術寄り、個人開発者・コミュニティ向け。

### Q-Mkt-05. 部分開示 vs 全面開示 の Owner 判断

- **背景**: `marketing-portfolio-reflection-design.md` §1（300 字結論）で「部分掲載（Conditional Partial Publish）」を採用。§1.1 公開可能 / 非公開境界表 + §5 必須非公開チェックリストで具体化。BAN 損失レンジ ¥500k〜¥2M / 12 ヶ月以内 BAN 確率 30〜60%、12h/$1,000 閾値、OAuth 経路、API キーフォールバック、mock-claude スタブ、ToS グレー解釈表現は非公開対象。同 §7-3 CEO 判断項目で「婉曲化方針の詳細」明示。
- **CEO 推奨案**: 部分開示維持。「商用 AI コーディングプラットフォーム」表記で固有名詞匿名化、BAN 確率 30〜60% は「重大運用リスクを定量評価して受容」へ婉曲化。CEO コメント: 「全面開示は他社実装の根拠化リスク + ToS 違反誘発で取り下げ確率激増」。
- **Owner 判断項目**: A) 部分開示で確定（Marketing 推奨）／ B) 全面開示（内部数値も含めて公開）／ C) 完全匿名化（PRJ 名すら出さない）／ D) 自由記述。
- **判断材料**:
  - A 案 pros: 差別化資産公開と ToS リスク回避の両立。cons: 内部数値非公開で訴求弱体化の懸念。
  - B 案 pros: 透明性最大化、コミュニティ評価。cons: 第三者が真似て ToS 違反 → 連鎖 BAN リスクで自社も巻き添え可能性。
  - C 案 pros: 法務リスクほぼゼロ。cons: ポートフォリオ訴求価値喪失、§3 LP 設計が成立しない。
- **回答が遅れた場合の影響**: §5.1〜§5.4 チェックリストの最終固定遅延 → Review 部門の二重チェック（公開前 GATE）を 6/12 までに通過できず 6/20 公開取りやめ。

### Q-Mkt-06. 自社 HP（COMPANY-WEBSITE）配置先 / リード獲得導線の Owner 確認

- **背景**: `marketing-portfolio-reflection-design.md` §1.2 掲載先選定表で `/works/clawbridge` + 技術ブログ 1 本 + About 1 段落の 3 点構成、`/services` および採用ページは不採用、プレスリリースも不採用。§3.2 S7 CTA で「お問い合わせ → /contact」「他の実績を見る → /works」導線。canonical URL `https://improver.jp/works/clawbridge`。
- **CEO 推奨案**: Marketing 提案 3 点構成のまま承認。CTA は /contact + /works 二択維持。CEO コメント: 「/services 配置は受託サービス誤認、採用は当面非活性のため不要」。
- **Owner 判断項目**: A) 3 点構成（Works 詳細 + 技術ブログ + About 追記）で承認／ B) 配置先追加・削除（指定）／ C) リード導線変更（メールフォーム以外、SNS DM / カレンダー予約等）／ D) 自由記述。
- **判断材料**:
  - A 案 pros: 既存 COMPANY-WEBSITE 構造変更最小、SEO 既存資産活用。cons: SNS 拡散導線弱い。
  - B/C 案: Owner の事業導線優先度に依存。/services への部分露出は受託案件への直接リード化の可能性あり、ただし「自社 PoC を受託サービスと混同される」誤解リスク。
- **回答が遅れた場合の影響**: COMPANY-WEBSITE Next.js 実装の `/works/clawbridge` ルート確定が遅れ、og:image / canonical / sitemap.xml 生成が連鎖遅延。

### Q-Mkt-07. Phase 1 完了時のプレスリリース or SNS 発信の有無

- **背景**: `marketing-portfolio-reflection-design.md` §1.2 掲載先選定表で「お知らせ / プレスリリース」=「採用しない」（理由: 「BAN リスク受容の PoC をプレス的に大きく出すのは事業方針『クリーンなデザイン』『AI 感を出さない』と齟齬」）。一方 §2.2 想定読者 D「AI / 開発者コミュニティ（HN / Zenn / X）」が 10% 比重で「拡散導線」として残存し、扱いが暗黙的に未確定。
- **CEO 推奨案**: プレスリリース不実施 + 公式 X / Zenn 軽度告知のみ（炎上回避のため「AI が AI を運営」煽り文 NG、harness engineering の知見公開という抑制トーン）。CEO コメント: 「プレス級は事業方針齟齬、ただし Zenn 技術記事拡散は採用見込み + 個人開発者層への到達に有効」。
- **Owner 判断項目**: A) プレス無し + Zenn / X 軽度告知（CEO 推奨）／ B) プレスリリース実施／ C) 全 SNS 発信なし（HP 内のみ）／ D) 自由記述。
- **判断材料**:
  - A 案 pros: 事業方針整合 + コミュニティ到達。cons: 拡散規模は予測不能。
  - B 案 pros: 媒体露出最大化。cons: BAN リスク受容 PoC が一人歩き、取り下げ時に風評リスク。
  - C 案 pros: 完全コントロール。cons: 想定読者 B/D（40%）への到達ゼロ、SEO 流入のみに依存。
- **回答が遅れた場合の影響**: 6/20 公開当日の発信プラン未定 → Web 運営 / Marketing の発信文面執筆が後ろ倒し、公開当日のチャネル統制不能。

### Q-Mkt-08. K8 / K9 anti-pattern の社外公開範囲（社内限定 vs 抽象化公開）

- **背景**: `marketing-knowledge-reflection-design.md` §2 K8 `ANTI-001-multi-account-tos-bypass.md`（multi-account 連鎖 BAN）/ K9 `ANTI-002-oauth-token-spawn-direct.md`（OAuth トークン直接到達）の 2 件は anti-pattern 性質上、内部数値や具体回避手順を含む。同 §7-5 CEO 判断項目で「K8 / K9 anti-patterns 公開範囲（社外共有時にも安全か、社内限定のみか）」明示。`marketing-portfolio-reflection-design.md` §5.1 で「OAuth 経路 / API キーフォールバック / multi-account 連鎖 BAN 再現条件」を必須非公開と既定済のため、K8/K9 公開可否が portfolio 反映と整合する必要あり。
- **CEO 推奨案**: 社内限定（`organization/knowledge/anti-patterns/` のみ配置、HP / 技術ブログには登場させない）。CEO コメント: 「両 anti-pattern は『してはいけない手順』を含むため公開すると逆効果、社内ナレッジ閉域化が安全」。
- **Owner 判断項目**: A) 社内限定（CEO 推奨）／ B) 抽象化して社外公開（具体手順を伏せて教訓のみ公開）／ C) 完全公開／ D) 自由記述。
- **判断材料**:
  - A 案 pros: ToS / BAN リスク完全遮断。cons: 社外への教訓共有はゼロ。
  - B 案 pros: 業界貢献。cons: 抽象化の度合い判断が困難、誤読されると逆効果。
  - C 案 pros: 透明性。cons: 第三者の ToS 違反誘発、Q-Mkt-05 部分開示方針と矛盾。
- **回答が遅れた場合の影響**: K8/K9 ファイル雛形の Phase 1 完了 KPT 起票が遅延、半年棚卸し（2026-12 末）対象外となり再評価フローに乗せられない。
