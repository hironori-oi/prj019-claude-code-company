## §7 R-019-04 cost overrun スコア再計算

### §7.1 R-019-04 内容（再掲）

| 項目 | 内容 |
|---|---|
| **ID** | R-019-04 |
| **内容** | cost overrun（月次予算超過、API 暴走 / drill 集中 / spike 等） |
| **登録時格付** | 黄（v1） |

### §7.2 スコア計算（v1 vs v2）

| 観点 | v1 | **v2** | 差分 | 根拠 |
|---|---|---|---|---|
| **確率** | **3**（高） | **2**（中低） | -1 | $300 cap → **$30 Hard cap で物理強制**、突破確率激減（5 spike scenario でも cap 突破は L-M 確率） |
| **影響** | **4**（高） | **3**（中） | -1 | 突破時も subscription $400 主軸は維持、Phase 1 全停止には至らない（API 直接消費機能のみ pause） |
| **スコア（確率 × 影響）** | **12（黄）** | **6（緑）** | **-6** | **黄 → 緑 改善** |

### §7.3 改善要因の詳細

| 要因 | v1 → v2 改善幅 |
|---|---|
| **物理 Hard cap による cap 突破の物理不可能化** | 確率 -1（Anthropic Console 側で API 強制停止） |
| **subscription only fallback 手順事前文書化**（§9.3） | 影響 -1（API 停止しても subprocess spawn は継続可能） |
| **二段警告体制**（Soft $25 + Hard $30） | 確率 -0.5（warn 段階で人間介入機会増加） |
| **subprocess kill 連動**（95% auto-stop = P-UI-04） | 影響 -0.5（kill switch で全体への波及防止） |
| **monitor cron 15min 高頻度**（Tracker §1.2） | 確率 -0.5（早期検知） |

合計改善: 確率 -1 / 影響 -1 = **スコア -6（12 → 6、黄 → 緑）**

### §7.4 結論

**R-019-04 は v2 で 黄 → 緑 に改善**。cap 強化と多層防御 (Soft + Hard + auto-stop + subscription fallback) により、cost overrun リスクは Phase 1 期間中に **低リスク** 水準まで抑制。
